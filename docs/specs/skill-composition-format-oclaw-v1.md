




OpenClaw 的 **Skills 组合是通过任务调度机制将多个 Skill 串联为自动化工作流的能力定义规则**，其核心在于 **明确组合逻辑、最小化依赖、保障容错性**，而非简单堆砌功能。**组合必须基于目标导向设计，避免冗余 Skill 干预流程，且关键步骤需内置异常处理机制**。以下为 2026 年 4 月最新版 OpenClaw 2.3+ 的官方组合规则：

---

## 一、组合类型与设计规则
### 1. **基础组合模式**
#### （1）串行组合（线性工作流）
- **定义规则**：  
  Skill 按 **严格顺序执行**，前序 Skill 的输出必须作为后序 Skill 的输入参数。  
  **必须声明数据传递格式**（如 JSON 路径、文件路径），禁止隐式依赖上下文。
- **示例**：  
  `web-clipper → content-extractor → text-summarizer → notion-sync`  
  （网页剪藏 → 提取正文 → 生成摘要 → 同步到 Notion）  
  需在组合描述中明确：`content-extractor.input = web-clipper.output.file_path`。

#### （2）并行组合（多路协同）
- **定义规则**：  
  多个 Skill **独立执行后汇总结果**，需定义 **结果合并逻辑**（如优先级规则、冲突解决策略）。  
  **禁止无限制并行**：并行 Skill 数量 ≤ 5，且需声明超时阈值（默认 30 秒）。
- **示例**：  
  同时调用 `tavily-search` 和 `brave-search` 获取信息后，通过 `content-analyzer` 比对可信度，**必须指定主数据源优先级**（如 `tavily-search` 权重 70%）。

#### （3）条件组合（动态分支）
- **定义规则**：  
  根据 **预设条件**（如输入参数、环境变量、前序结果）动态选择 Skill 路径。  
  **必须覆盖所有分支场景**，包括默认回退路径（如条件不匹配时触发 `fallback-skill`）。
- **示例**：  
  ```markdown
  if 用户查询含“股票” → 调用 `stock-analyzer`  
  else if 含“天气” → 调用 `weather`  
  else → 触发 `summarize`（默认路径）
  ```
  条件判断逻辑需在组合描述中 **用伪代码明确定义**。

## 二、组合设计核心原则
### 1. **目标导向（强制）**
- **禁止冗余 Skill**：  
  组合中每个 Skill 必须直接贡献于最终目标。例如生成热点简报时，**无需接入翻译 Skill**（除非用户明确要求多语言输出）。
- **输入/输出对齐**：  
  前序 Skill 的输出格式 **必须严格匹配** 后序 Skill 的输入要求（如 `text-summarizer` 需接收 `content-extractor` 生成的 `text/plain` 格式）。

### 2. **最小依赖（强制）**
- **轻量级组合**：  
  单个工作流 Skill 数量 **建议 ≤ 7 个**，超过需拆分为子组合（如 `数据预处理子流` + `报告生成子流`）。
- **依赖显式声明**：  
  组合需在元数据中声明 **跨 Skill 依赖项**（如 `requires: ["content-extractor@1.2.0+"]`），避免版本冲突。

### 3. **容错冗余（强制）**
- **关键节点必须配置备选路径**：  
  例如网络请求类 Skill 需绑定重试策略（`max_retries: 3`）或备选 Skill（如 `tavily-search` 失败时切换 `brave-search`）。
- **超时与熔断机制**：  
  单个 Skill 执行超时默认 **30 秒**，组合级超时需显式定义（如 `workflow_timeout: 120s`），超时后自动触发 `error-handler`。

## 三、组合定义与执行规范
### 1. **组合声明方式**
#### （1）隐式组合（通过用户指令）
- 用户直接描述任务流程，由 OpenClaw **自动解析 Skill 依赖链**：  
  `“先搜索 AI 趋势，再总结核心观点，最后生成 PPT 大纲”`  
  → 自动匹配 `web-search → text-summarizer → ppt-outline`。
- **限制**：仅支持线性流程，**不保证复杂条件的准确性**。

#### （2）显式组合（通过组合 Skill）
- 创建 **专用组合 Skill**（如 `daily-report`），在 `SKILL.md` 中明确定义流程：  
  ```markdown
  ## 执行步骤
  1. 调用 `calendar` 获取日程 → 输出 `schedule.json`
  2. 调用 `weather` 查询天气 → 输出 `weather.json`
  3. 调用 `docx` 生成报告 → 输入 `schedule.json` + `weather.json`
  ```
  **必须标注数据传递路径**，否则智能体无法关联上下文。

### 2. **执行机制**
- **动态加载**：  
  仅当用户触发组合目标时，**按需加载关联 Skill**，避免启动时加载全部能力。
- **状态隔离**：  
  每个组合运行 **独立会话上下文**，前序 Skill 的中间结果 **仅对当前组合可见**，防止跨任务污染。

## 四、验证与调试规则
### 1. **组合健康检查**
- **必须通过三类测试**：  
  - **结构验证**：检查 Skill 依赖链是否断裂（`openclaw skills validate-combo daily-report`）。  
  - **边界测试**：模拟空输入、超长文本、API 失效等异常场景。  
  - **性能压测**：单次组合执行时间 **不得超过 2 分钟**（默认超时阈值）。

### 2. **日志与回溯**
- **强制记录关键节点**：  
  每个 Skill 的输入/输出、执行状态（成功/失败）、耗时必须写入审计日志。  
- **支持流程回放**：  
  通过 `openclaw workflow replay <session-id>` **重现完整执行链**，定位故障点。

---

## 五、典型错误规避
### 1. **常见违规示例**
- **隐式上下文依赖**：  
  错误写法：`“用上一步的结果生成报告”` → 正确写法：`“输入 text-summarizer.output.summary_text”`。  
- **无熔断的循环调用**：  
  Skill A 调用 Skill B，而 Skill B 又触发 Skill A → **必须设置最大递归深度（默认 5 层）**。

### 2. **生产环境限制**
- **禁止跨账户组合**：  
  组合内所有 Skill **必须来自同一信任域**（如同一工作区或同一权限组）。  
- **敏感操作需人工确认**：  
  涉及数据删除、外部系统写入的组合，**必须插入用户确认节点**（如 `“确认执行删除？[Y/N]”`）。

> 💡 **关键提示**：技能组合的 **核心价值在于将经验固化为可复用流程**，而非功能叠加。设计时需优先验证 **单点可靠性**（每个 Skill 独立通过测试），再构建组合逻辑。生产环境建议通过 `clawhub combo publish` 发布组合，并附带 **完整异常处理路径说明**。
